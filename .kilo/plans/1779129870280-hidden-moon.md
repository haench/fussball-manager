# Plan: Transfermarkt Game Loop — Fussball Manager

---

## UI Flow

Screens in order of appearance:

1. **Club Hub** → neue Kachel **Transfermarkt**
2. **Transfermarkt** → zwei Tabs horizontal: **Kaufen** | **Verkaufen**
   - Unterhalb beider Tabs: eine **Logzeile** mit den letzten Ereignissen (Käufe, Verkäufe, Angebot erhalten und bearbeitet)
3. **Kaufen-Tab** → Liste von Spielern anderer Vereine der eigenen Liga, max. 5 pro Spieltag neu generiert
4. **Verkaufen-Tab** → ähnlich Kader-View: Spieler eigener Mannschaft aufgeteilt nach **Startelf** und **Bank**; jedes Spielerzeile enthält Knopf **"Verkaufen"**
5. **Externe Angebote** erscheinen in der gemeinsamen Logzeile; können sofort angenommen oder abgelehnt werden

---

## Current "Game Loops" Identified

Implementierte, bereits vorhandene Schleifen, nach deren Muster gearbeitet wird:

| Loop | Datei | Muster |
|---|---|---|
| Match-Simulation | `gameLoop.js` `runMatchLoop()` | Aktives Intervall (`setInterval` 33 ms, fixed-timestep Akkumulator) |
| Post-Match-Phase | `gameLoop.js` `continueAfterMatch()` | Einmaliger Aufruf nach Spielende, ruft nacheinander `processTrainingAfterMatch`, `processYouthAfterMatch`, `progressStadiumUpgrades` |
| Tages-Progression | `leagueWorld.js` `completeMatchDay()` | Einmaliger Aufruf pro Spieltag in Ligawelt |
| Training-Upgrade | `progressTrainingFacilityUpgrade()` | Dekrementiert `daysRemaining` pro Tag, schließt bei 0 ab |
| Jugend-Upgrade | `progressYouthAcademyUpgrade()` | Gleiches Muster |
| Stadion-Upgrade | `progressStadiumUpgrades()` | Gleiches Muster |

**Transfermarkt ist KEINE weiterlaufende aktive Schleife.** Sie ist ein **zustandsgesteuerter, einzelaufruf-artiger Mechanikblock**, der in der Post-Match-Phase aufgerufen wird — exakt wie Jugend, Training und Stadion-Upgrades.

## Neue Daten

### Team-Feld (neu)

```js
transferMarket: {
  externalOffer: null | {
    id: string;               // eindeutige Angebots-ID
    buyerClubName: string;    // z. B. "SV Grün-Weiß"
    playerName: string;       // Name des zum Verkauf stehenden Spielers
    offerAmount: number;      // gebotener Preis, ~80 % des Marktwerts
    createdDay: number;       // Spieltag, an dem das Angebot erstellt wurde
    expiresDay: number        // Spieltag, an dem das Angebot ungültig wird
  }
}
```

### Spieler-Felder (neu, pro Spieler)

```js
marketValue: 50000;        // berechnet als strength² × 100, gerundet auf 1000
isOnTransferList: false;   // Spieler steht auf der Verkaufsliste
```

Preisformel:
```
marketValue = Math.round(strength * strength * 100 / 1000) * 1000
```
Später ligaabhängig.

## Logzeile (Ereignisanzeige)

Die Transfermarkt-Ansicht zeigt immer eine Logzeile unterhalb der Tab-Inhalte mit maximal **5 Einträgen**, neueste zuerst. Eintragstypen:

| Eintragstyp | Beispieltext |
|---|---|
| `bought` | `Max Mustermann für 12.000 € gekauft.` |
| `listed` | `Tim Becker steht jetzt zum Verkauf.` |
| `unlisted` | `Tim Becker wieder von der Liste genommen.` |
| `external_offer` | `SV Grün-Weiß bietet 80.000 € für Tim Becker.` |
| `accepted` | `Angebot für Tim Becker von SV Grün-Weiß angenommen (80.000 €).` |
| `rejected` | `Angebot für Tim Becker abgelehnt.` |
| `expired` | `Angebot für Tim Becker ist abgelaufen.` |

Log-Daten: `gameState.transferMessages` — Array von maximal 5 Einträgen `{ key, text }`. Wird in Thread-Local Storage persistiert. Alte Einträge unten abschneiden, neue oben (`unshift`).

## Externe Angebote — Detail

- Im Post-Match-Loop prüft die Funktion `processTransferMarketLoop(state)` pro Spieltag mit **10 % Wahrscheinlichkeit**, ob ein Angebot generiert wird.
- Angebot wird erstellt, falls `gameState.transferMarket.externalOffer` nicht existiert (es gibt höchstens ein aktives Angebot gleichzeitig).
- Käufer-Club: zufälliger **verein des gleichen Ligazweigs** (nur echte Liga-Clubs, keine Reserve-Teams).
- Angebotsbetrag: **Marktwert × 0,8**, immer auf 1000 gerundet.
- Angebot ist **1 Spieltag gültig** `expiresDay = state.currentDay + 1` → wird am Folgetag automatisch verworfen und als `expired`-Eintrag in die Log geschrieben.
- Angebot erscheint sofort als neuester Eintrag in der Logzeile sowohl im **Kaufen**- als auch im **Verkaufen**-Tab.
- Annahme führt zu: Geld wird gutgeschrieben, Spieler wird aus dem Kader entfernt, `financeLedger`-Eintrag als `income / transfer`, Angebot wird gelöscht, `accepted` in Log.
- Ablehnung oder Ablauf: Angebot wird gelöscht, Spieler bleibt im Kader, `rejected` bzw. `expired` in Log.

## Liga-Struktur — woher kommen Käufer-Clubs

`leagueWorld.js` hat eine Dreiliga-Struktur (`liga1`, `liga2`, `liga3`), je 18 Teams. Die `games.teams` enthält alle vereine, wobei der Benutzer ein spezifisches Team aus `liga3` hat. Käufer-Clubs werden aus allen Teams desselben Ligazweigs gezogen (also alle `liga1`-Teams, falls der Benutzer in liga1 spielt) — ohne Reserve-Teams. Angebote aus höheren Ligen sind durchweg unwahrscheinlich, niedrigeren Ligen hohe Wahrscheinlichkeit — dies wird für spätere Iteration reserviert, aktuell nur gleiche Liga.

## Storage-Compatibilität

Es gibt eine bestehende Validierungsfunktion `storage.js` `isPlayer(value)`, die nur bekannte Felder prüft. Alte gespeicherte Spieler haben noch keine `marketValue` und `isOnTransferList`. Beim `normalizeSaveData()`-Aufruf müssen diese fehlenden Felder mit Fallback-Werten (`marketValue: 0`, `isOnTransferList: false`) belegt werden, damit keine Alt-Saves ungültig werden.

## Geplante Integration in Post-Match-Phase

`gameLoop.js` Funktion `continueAfterMatch()` erst die bereits vorhandenen Blockaufrufe (Stadion-Upgrade ausführen), dann neu **`processTransferMarketLoop()`** — das ist der einmalige, nicht-endliche Aufruf, sobald ein Spiel vorbei ist. Danach wird die Transfermarkt-Ansicht durch die Ermittlung von `gameState.transferMarket.externalOffer` ggf. ein Angebot anzeigen, bis der Spieler darauf reagiert oder es verfällt (`continueAfterMatch()` bei der nächsten Spieltag-Ausführung räumt es auf).
